PHASE 1 — ANFORDERUNGSANALYSE
================================================================================

1.1 Personas
-------------
Persona H (Hauptnutzer / Entwickler):
- Benoetigt Dashboard fuer Pipeline-Visualisierung und -Analyse
- Will Native Win App (mainly), waere gut auch Mac
- Will Zeitsparen bei Verbesserung der Agenten-Pipeline
- Dashboard soll schnell sein und visuell gut aussehen
- Terminal-Integration: Ausfuehrung in der App sehen/steuern
- OAuth Token von verschiedenen Anbietern speichern
- Powershell/Bash Instanzen starten, Input/Output sehen/steuern
- Arbeit mit Agenten/Skills/Hooks deterministischer machen
- GitHub-Status pro Projekt: PRs, Issues, Branch, letzter Commit (2026-03-17)
- Agenten-Konfiguration pro Projekt einsehen: CLAUDE.md, Skills, Hooks, MCP (2026-03-17)

Persona C (Co-Nutzer / AI-Hub-User):
- Funktionen per Chat ueberpruefen, Schnittstellen zu Anbietern (Kalender, Midjourney)
- Dashboard als zentraler Hub fuer AI- und arbeitsverbundene Aufgaben
- API-Anfragen und Kosten ueberwachen pro API Key
- GitHub-Projekte: Fortschritt anzeigen
- Andere AI-Projekte vom Dashboard aus starten

1.2 User Stories
-----------------

--- Release v1.0.0 (ERLEDIGT — 2026-03-17) ---

US-S1: [DONE] Als Entwickler (H) will ich mehrere Claude CLI Sessions in einem Fenster verwalten.
US-S2: [DONE] Als Entwickler (H) will ich Sessions erstellen, umbenennen, wechseln und schliessen.
US-S3: [DONE] Als Entwickler (H) will ich Projektordner pro Session auswaehlen und zuweisen.
US-S4: [DONE] Als Entwickler (H) will ich in der Sidebar erkennen ob eine Session aktiv arbeitet, denkt, wartet oder fertig ist.
US-S5: [DONE] Als Entwickler (H) will ich globale Notizen im Dashboard festhalten.
US-S6: [DONE] Als Entwickler (H) will ich die aktive Session mit Projektname im Header sehen.

--- Release v1.1.0 (GEPLANT) — Agenten-Transparenz ---
Prioritaet: HOCH | Aufwand: MITTEL | Abhaengigkeiten: Tauri File-Read Command

US-A1: Als Entwickler (H) will ich die CLAUDE.md eines Projekts im Dashboard lesen koennen,
       damit ich die Agenten-Anweisungen pruefen kann ohne die Datei manuell zu oeffnen.
       Akzeptanzkriterien:
       - CLAUDE.md wird aus dem Session-Ordner geladen
       - Markdown wird gerendert dargestellt
       - Aktualisiert sich bei Session-Wechsel

US-A2: Als Entwickler (H) will ich die registrierten Skills (.claude/skills/) eines Projekts
       auflisten und einsehen, damit ich nachvollziehen kann welche Faehigkeiten die Agenten haben.
       Akzeptanzkriterien:
       - Alle .md Dateien in .claude/skills/ werden aufgelistet
       - Klick auf einen Skill zeigt den Inhalt
       - Leerer Zustand wenn kein .claude/skills/ Ordner existiert

US-A3: Als Entwickler (H) will ich die Hooks-Konfiguration (.claude/settings.json) eines Projekts
       sehen, damit ich verstehe welche automatischen Aktionen bei Events ausgeloest werden.
       Akzeptanzkriterien:
       - settings.json wird geladen und Hooks-Sektion wird strukturiert angezeigt
       - Unterscheidung zwischen Projekt- und User-Settings
       - Fehlerfall: "Keine Hooks konfiguriert" wenn Datei fehlt

US-A4: Als Entwickler (H) will ich nach einer Session pruefen koennen ob die Agenten sich an
       das vorgesehene Design gehalten haben — eine Art Audit-View.
       Akzeptanzkriterien:
       - Zusammenfassung: welche Skills wurden aufgerufen vs. welche existieren
       - Abweichungen vom CLAUDE.md hervorheben (Stretch Goal, v1.2.0+)

--- Release v1.2.0 (GEPLANT) — GitHub-Integration ---
Prioritaet: MITTEL | Aufwand: HOCH | Abhaengigkeiten: GitHub Token oder gh CLI

US-G1: Als Entwickler (H) will ich pro Session/Projekt die offenen PRs sehen,
       damit ich den Review-Status kenne ohne den Browser zu oeffnen.
       Akzeptanzkriterien:
       - Offene PRs mit Titel, Autor, Status (draft/review/approved)
       - Klick oeffnet PR im Browser

US-G2: Als Entwickler (H) will ich pro Session/Projekt die offenen Issues sehen,
       damit ich weiss welche Bugs/Tasks anstehen.
       Akzeptanzkriterien:
       - Offene Issues mit Titel, Labels, Assignee
       - Filterbar nach Label

US-G3: Als Entwickler (H) will ich den aktuellen Branch und letzten Commit pro Projekt sehen,
       damit ich weiss auf welchem Stand jede Session arbeitet.
       Akzeptanzkriterien:
       - Branch-Name + letzter Commit (Hash, Message, Zeitpunkt)
       - Aktualisiert sich automatisch (Polling oder Event-basiert)

1.3 Machbarkeitsanalyse
-------------------------

v1.0.0 Session Manager: [ERLEDIGT] — Tauri v2 + React + PTY = funktioniert produktiv.

v1.1.0 Agenten-Transparenz:
- CLAUDE.md lesen: Machbar. Neuer Tauri-Command `read_file(path)` noetig.
  Tauri v2 erlaubt Filesystem-Zugriff. Kein Security-Risiko da nur Lesen im Projektordner.
- Skills auflisten: Machbar. `read_dir` + `read_file` fuer .claude/skills/*.md.
- Hooks anzeigen: Machbar. JSON-Parse von .claude/settings.json im Frontend.
- Audit-View: Teilweise machbar. Einfache Version (Config anzeigen) ja.
  Volle Audit (was wurde wirklich aufgerufen) braucht Log-Analyse — Stretch Goal.
- Fazit: MACHBAR mit bestehendem Tech-Stack, kein neues Tooling noetig.

v1.2.0 GitHub-Integration:
- Branch/Commit: Machbar. `git` CLI Commands via Tauri Shell-Plugin.
- PRs/Issues: Machbar. Entweder `gh` CLI oder GitHub REST API direkt.
  Token-Verwaltung existiert bereits im settingsStore (apiKeys[]).
- Fazit: MACHBAR, braucht GitHub Token Onboarding im Settings-UI.
